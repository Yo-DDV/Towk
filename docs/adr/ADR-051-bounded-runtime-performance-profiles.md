# ADR-051: Bounded Runtime Performance Profiles Stay Below Operator Envelopes

**Date:** 2026-07-17

## Context

Towk runs on hosts that range from small community VPS instances to larger
dedicated servers. The deployment currently carries independent static limits
for Towk, NATS, LiveKit, Caddy, image transforms, link previews, video workers,
and realtime connections. Raising one of those values in isolation can move the
bottleneck or increase memory pressure without increasing useful throughput.

An owner-facing performance selector is useful only if it reports what the
process can actually apply. It must not imply that a browser can resize a
container, alter an environment file, restart a service, or override an
operator's safety boundary.

Go 1.25 and later make the default `GOMAXPROCS` container-aware, but the Go
runtime deliberately rounds fractional CPU quotas up and normally does not
select a value below two solely from a cgroup quota. Towk therefore treats
`GOMAXPROCS` as one input, not as proof that every requested worker can run in
parallel. Docker Compose `cpus`, `mem_limit`, and `mem_reservation` remain
deployment-level controls. JetStream object data must still fit on the target
filesystem and is not a substitute for distributed object storage.

## Decision

Towk has two explicit capacity layers:

1. **Operator envelope.** Compose/cgroup CPU and memory limits, storage quotas,
   process configuration, and optional per-worker caps are deployment inputs.
   Changing them may require recreating or restarting a service.
2. **Runtime policy.** An owner can request `economy`, `balanced`,
   `performance`, or bounded `custom` concurrency through the admin API. The
   runtime derives effective values below the operator envelope and current
   process capacity.

The browser never receives a Docker socket and never writes Compose, TOML, or
environment files. The admin response exposes requested and effective values,
the source of every cap, the detected process envelope, and whether a related
operator change needs a restart.

### Profile persistence and compatibility

The requested policy is a versioned event-sourced server configuration fact.
It carries a revision used for optimistic concurrency. A stale write is
rejected instead of silently replacing a newer owner's choice.

New configurations generated by `towk init` request `balanced`. An existing
installation with no stored policy keeps the historical worker values. The
balanced preset is initially calibrated to those historical values, so applying
it does not silently increase upgraded installations.

Unknown profile names and unsupported schema versions are configuration errors.
The current process falls back to the conservative economy limits and exposes
the error to owners; it does not guess a future profile. Older binaries ignore
the new event type and therefore require the documented operator caps during a
binary rollback.

### Precedence

For every runtime limit, the effective value is the minimum applicable value in
this order:

1. detected process capacity (CPU affinity/cgroup quota and memory limit);
2. explicit operator cap;
3. requested preset or custom value;
4. historical value when no policy exists, otherwise the balanced value for a
   new generated configuration;
5. a conservative built-in fallback when inputs are invalid.

Zero is never interpreted as unlimited for a worker count. Custom values have
server-side minima and maxima. The implementation retains the reason for every
reduction so the UI can explain it instead of showing a misleading preset badge.

### Live changes

Runtime worker admission reads the projected policy. Lowering a limit never
cancels work that was already admitted. New work waits or is rejected according
to the existing subsystem contract until active work is below the new limit.
Raising a limit can admit later work without a process restart.

Infrastructure values such as container memory, container CPU, NATS resources,
and process startup-only listeners remain operator-owned and are marked as
restart-required. The UI may show guidance for them but cannot mutate them.

### Authorization and audit

Reading or changing performance settings requires the effective server owner
role. Validation and authorization are enforced in core before mutation. The
durable event records the actor and bounded policy values, but no host path,
secret, environment content, or deployment identifier.

## Consequences

- Small hosts receive conservative concurrency without a separate deployment
  stack.
- Larger hosts can use more workers only when both the requested policy and
  operator envelope allow it.
- A profile name is a policy, not a capacity or user-count guarantee. Published
  capacity claims require reproducible qualification on the stated CPU,
  memory, storage, architecture, and network envelope.
- Multi-replica deployments share the requested event-sourced policy while each
  process derives its own effective values from its local envelope.
- The implementation must test upgrades without a policy, concurrent writes,
  invalid/future policies, live downgrade under load, restart, rollback, and
  heterogeneous process envelopes.
