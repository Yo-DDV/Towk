Towk supports real-time voice and video calls inside rooms using [LiveKit](https://livekit.io/), an open-source WebRTC media server. When enabled, any room member can start or join a call directly from the room header.

## How It Works

LiveKit runs as a **separate service** alongside Towk. Towk handles authentication and call state, while LiveKit handles the WebRTC media transport:

1. A user joins the call from the room header.
2. Towk verifies the user's room membership and generates a short-lived LiveKit JWT token.
3. The user's browser connects directly to the LiveKit server using the token and a per-call E2EE key from Towk.
4. Other room members see active call indicators update in real time.

Calls are **implicit** — there's no "start call" action. The first person to join creates the call and a fresh E2EE key; when the last person leaves, the call ends and Towk shreds that key.

Towk records call start/join/leave/end facts durably and reconciles active participants against LiveKit webhooks, so call indicators recover after server restarts. A recoverable media disconnect marks the affected connection as interrupted for a bounded grace period instead of immediately removing it; explicit leaves and moderation removals remain terminal. These call facts update the active call UI without appearing as room timeline messages. LiveKit media is end-to-end encrypted by current Towk clients; older clients that do not support this call encryption cannot decode encrypted media.

## Requirements

- A running [LiveKit server](https://docs.livekit.io/home/self-hosting/local/) (self-hosted or cloud)
- An API key and secret configured on both the LiveKit server and Towk

## Setup

<Steps>

1. **Install and run LiveKit**

   For local development, the quickest option is the LiveKit CLI:

   ```bash
   # macOS
   brew install livekit

   # Then run in dev mode
   livekit-server --dev
   ```

   This starts LiveKit on `ws://localhost:7880` with a default API key of `devkey` and secret of `secret`.

   For production, see the [LiveKit self-hosting guide](https://docs.livekit.io/home/self-hosting/local/).

2. **Configure Towk**

   Add a `[livekit]` section to your `towk.toml`:

   Development:

   ```toml
   [livekit]
   enabled = true
   url = 'ws://localhost:7880'
   api_key = 'devkey'
   api_secret = 'secret'
   ```

   Production:

   ```toml
   [livekit]
   enabled = true
   url = 'wss://livekit.example.com'
   api_key = 'your-api-key'
   api_secret = 'your-api-secret'
   # webhook_url = 'https://towk.example.com/webhooks/livekit'
   # defaults to {webserver.url}/webhooks/livekit
   ```

   Or use environment variables:

   ```bash
   CHATTO_LIVEKIT_ENABLED=true
   CHATTO_LIVEKIT_URL=wss://livekit.example.com
   CHATTO_LIVEKIT_API_KEY=your-api-key
   CHATTO_LIVEKIT_API_SECRET=your-api-secret
   # CHATTO_LIVEKIT_WEBHOOK_URL=https://towk.example.com/webhooks/livekit  # defaults to {webserver.url}/webhooks/livekit
   ```

3. **Configure LiveKit webhooks**

   LiveKit must send webhook events to Towk so it can track who's in a call. Add the webhook URL to your LiveKit server config (`livekit.yaml`):

   ```yaml
   webhook:
     urls:
       - https://towk.example.com/webhooks/livekit
     api_key: your-api-key
   ```

   For local development with `livekit-server --dev`, the `--dev` flag disables webhooks. That is useful for testing local media connectivity, but active-call indicators and participant reconciliation require webhook delivery in normal deployments.

4. **Restart Towk**

   Calls will be available immediately. Call controls appear in room headers when LiveKit is configured.

</Steps>

## Production Considerations

:::note
Use `wss://` (WebSocket Secure) URLs in production. Plain `ws://` should only be used for local development.
:::

- **Network**: Users connect directly to the LiveKit server from their browsers. Make sure your LiveKit server is reachable from the public internet (or your private network, for internal deployments). HTTPS/WebSocket signaling and WebRTC media use different routes, as described below.
- **TURN/STUN**: LiveKit includes a built-in TURN server for NAT traversal. TURN/UDP on port 3478 helps many NAT traversal failures; networks that block UDP entirely need TURN/TLS, often on port 443.
- **Scaling**: A single LiveKit server handles hundreds of concurrent participants for typical small deployments. For larger deployments, LiveKit supports multi-node clustering with Redis.
- **End-to-end encryption**: Towk enables LiveKit E2EE automatically for clients. No additional LiveKit server configuration is required.
- **API Secret**: Keep your API secret confidential. It's used to sign JWT tokens and should never be exposed to clients.

### Network Paths and Ports

Put LiveKit's API and WebSocket signaling endpoint behind a reverse proxy with
a publicly trusted certificate. Route `wss://livekit.example.com` on TCP 443 to
LiveKit port 7880. Media travels directly to LiveKit instead of through the HTTP
reverse proxy:

| Public port          | Purpose                                                         |
| -------------------- | --------------------------------------------------------------- |
| TCP 443              | Secure LiveKit API and WebSocket signaling, proxied to TCP 7880 |
| TCP 7881             | ICE/TCP media fallback when UDP is unavailable                  |
| UDP 3478             | Optional embedded TURN/STUN relay                               |
| UDP media range      | Direct WebRTC media; the Compose example uses 50000-50200       |
| UDP TURN relay range | TURN allocation sockets; the Compose example uses 50201-50400   |

The port range must match `rtc.port_range_start` and `rtc.port_range_end` in
your LiveKit configuration. Do not route the TCP or UDP media ports through a
normal HTTP reverse proxy.

  The [Docker Compose guide](/guides/deployment/docker-compose/) provides a
  complete working LiveKit configuration, firewall list, automatic HTTPS, and
  generated matching secrets.
</Aside>

## TURN for Restrictive Networks

LiveKit can usually connect browsers directly over UDP. Users behind symmetric NATs, mobile networks, or browser/network combinations that fail direct ICE checks need a TURN relay.

For most self-hosted deployments, enable LiveKit's built-in TURN/UDP server first:

```yaml
turn:
  enabled: true
  udp_port: 3478
  relay_range_start: 50201
  relay_range_end: 50400
```

Expose UDP 3478 on the LiveKit host, keep the UDP media range open, and publish
the configured TURN relay allocation range one-to-one when LiveKit runs behind
a container bridge. The listener alone is not the relay data path. Then enable
ICE/TCP on port 7881 as a fallback when UDP is unavailable. Networks that also
block this TCP fallback need TURN/TLS. If your users can only reach outbound TCP
443, LiveKit or TURN needs a dedicated IP address or host so it can listen on
443 without conflicting with your HTTPS reverse proxy, and the TURN TLS
certificate must match the advertised TURN domain.

For a public LiveKit node, prefer `rtc.use_external_ip: true`. When a custom
topology sets `use_external_ip: false` and `rtc.node_ip` is private, LiveKit's
embedded TURN permission handler otherwise rejects that private media peer.
Allow only the exact node address:

```yaml
rtc:
  use_external_ip: false
  node_ip: 10.0.0.10
turn:
  enabled: true
  udp_port: 3478
  allow_restricted_peer_cidrs:
    - 10.0.0.10/32
```

Use `/32` for one IPv4 node or `/128` for one IPv6 node. In a cluster, list the
individual restricted SFU addresses that embedded TURN must reach. These are
media-peer addresses, not caller networks: never add broad private ranges just
because callers use mobile, office, or home NATs.

Use a dedicated TURN server such as [coturn](https://github.com/coturn/coturn) when you need tighter network separation, a shared TURN service for multiple LiveKit deployments, or TURN/TLS on port 443 without moving LiveKit itself. Configure static credentials on the TURN server and list it under LiveKit's `rtc.turn_servers` configuration.

## Media and Resilience Behavior

Towk configures call media for graceful degradation rather than a fixed quality:

- **Microphone audio** requests mono capture with browser echo cancellation,
  noise suppression, voice isolation, and automatic gain control, without
  forcing a sample rate. The browser and operating system therefore retain the
  clock required by the active built-in, wired, Bluetooth HFP, or BLE route.
  Browser-native processing is preferred. The call menu lets users disable or
  re-enable noise reduction, automatic gain, and echo cancellation per call, and
  shows whether the active route reports native/system processing or a reduced
  capability. Towk adds the local RNNoise or SpeexDSP fallback only when the
  final track explicitly reports native noise suppression disabled and its
  source clock matches the processing context. Mobile browsers remain
  native-only for local processing so Android and iOS Bluetooth or built-in
  routes are not put behind a second Web Audio clock. If that fallback cannot
  start, the microphone remains live with browser audio processing. Opus DTX and
  RED remain enabled to reduce silence bandwidth and protect speech on lossy
  links.
- **Camera video** requests a 4:3 1280 × 960 stream at no more than 30 FPS and
  asks compatible browsers not to crop the source frame. This keeps the field of
  view of 4:3 phone cameras instead of selecting a tighter 16:9 sensor mode,
  while still publishing adaptive layers for smaller viewers and weaker links.
  LiveKit simulcast, dynacast, and adaptive receive selection let each viewer
  use the layer that fits their element and bandwidth; camera publication
  preserves frame rate first when congestion requires lower detail. Camera and
  screen content also use aspect-ratio-preserving rendering.
- **Screen sharing** publishes a 1080p/30 ceiling with 360p and 720p simulcast
  layers. The receiver can move between layers without stretching or cropping
  the shared desktop, while congestion adaptation preserves resolution first
  so shared text remains readable. Screen audio is included only when the
  browser and selected capture source provide it.
- **Codec negotiation** is left to WebRTC and LiveKit so participants receive a
  mutually supported codec. Do not force H.264 as the only codec: VP8 is the
  common interoperable baseline for the currently supported browser engines.

When signaling or media is interrupted, the local client shows a reconnecting
notice and preserves its device choices while LiveKit attempts recovery. If a
fresh connection is required, Towk revalidates the same call generation before
issuing another token, preventing a delayed request from joining or leaving a
replacement call. Microphone restoration first retries the selected route,
then an available input, then the operating-system route without a fixed device
ID if a mobile browser briefly exposes no stable input during a Bluetooth or
network transition. Other participants retain an interrupted participant card
during the recovery window and can see packet-loss or jitter degradation for
individual connections. A connection that does not return before the grace
period is removed by reconciliation.

## Call Controls and Devices

- **Microphone and incoming-audio controls** — Mute the microphone or all remote
  microphone and screen-share audio independently. The device menu also exposes
  per-call noise reduction, automatic gain, and echo-cancellation preferences
  with the current route status.
- **Audio and video device selection** — Switch microphones, speakers, and
  cameras without leaving. Towk restores the selected devices after call
  recovery when the browser still exposes them, serializes route changes,
  follows LiveKit's active device after automatic fallback, and replaces
  disappeared selections. If a microphone disappears while muted, Towk selects
  an available input before retrying unmute. If the browser keeps the logical
  `default` ID unchanged or returns an empty inventory during the route change,
  Towk drops the stale exact constraint and reacquires the route selected by the
  operating system. If native recovery has to mute a failed route, Towk shows a
  dedicated notice and retries after 1, 2, and 4 seconds; a detected device
  change retries immediately, while the bounded timer covers mobile browsers
  without `devicechange`. An explicit mute cancels recovery. Permission-denied
  and device-in-use failures remain visible and are not silently rerouted. The
  already active route is not restarted when selected again. Selecting a
  different microphone while recovery is visible immediately retries capture
  on that route, even after the bounded background schedule has ended. Changing
  a processing preference while the microphone is live reacquires capture
  through the same serialized queue. The menu re-enumerates devices when opened,
  uses the standards speaker picker where available, explains system-managed
  output on mobile, and replaces generic platform labels such as speakerphone,
  earpiece, or Bluetooth headset with localized route names while preserving
  useful desktop hardware names. Mobile lenses are presented as
  front/rear/ultra-wide/telephoto where labels permit, with a one-touch lens
  switch when several are exposed.
- **Camera and screen sharing** — Toggle either source independently, expand
  remote media without cropping, and inspect screen-share receive diagnostics.
- **Per-participant status** — Speaking level, connection quality, packet loss,
  jitter, interrupted state, camera, and screen-share state are shown on the
  relevant participant card.
- **Multiple devices** — One account can keep up to two call connections. The
  second can join as a muted companion or transfer the call; sibling devices can
  control each other's microphone and incoming-audio mute state.
- **Installed PWA integration** — Active and recovering calls retain the wake
  lock and Media Session controls. A validated cross-room or cross-server switch
  releases old local capture before the replacement connection starts.
- **Active call indicators** — Rooms with active calls show the current call and
  projected participants even before the viewer joins.

## Browser and Device Compatibility

Call entry fails closed when the browser cannot provide the required LiveKit
E2EE support. Camera and microphone permissions, audio-output switching, screen
capture, and system-audio capture remain browser and operating-system
capabilities: Towk detects unavailable screen sharing, reports media permission
or device failures, and keeps a call joined muted when microphone startup alone
fails. Bluetooth transport and communication-profile selection remain native
operating-system responsibilities; Towk follows the resulting WebRTC route and
does not infer profiles from localized labels. On current Android and iOS web
engines, installed PWAs do not receive the
native screen-capture API, so Towk disables that control and explains the
desktop alternative instead of promising a picker that cannot open. Screen
capture on supported desktop browsers always requires a user gesture and the
native picker may offer different sources on each platform.

For production acceptance, test the installed PWA on the browser, operating
system, and physical audio/video devices used by your organization. Include a
direct UDP path, ICE/TCP fallback, TURN/UDP, a real network handoff, device
permission denial, camera/microphone removal, and the platform's native screen
and system-audio picker. Browser simulation cannot replace those device-level
checks.

## Disabling Voice & Video Calls

To disable voice and video calls, either remove the `[livekit]` section from `towk.toml` or set:

```toml
[livekit]
enabled = false
```

When disabled, call controls are hidden. Call APIs return their normal
not-configured results: lists are empty, join and leave report `false`, and a
singular active-call lookup returns `NOT_FOUND`.
