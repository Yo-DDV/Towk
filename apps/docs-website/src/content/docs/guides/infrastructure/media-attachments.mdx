Towk can handle ordinary file uploads, images, avatars, server branding images, link-preview images, and optionally videos and animated GIFs. Small deployments can keep assets in NATS ObjectStore; larger deployments should usually move assets to S3-compatible storage.

## What Members Can Upload

Members can attach files through drag-and-drop, paste, or the file picker when they have `message.attach` and the relevant posting permission.

On Windows, macOS, and Linux, Towk accepts every file object the browser exposes from the native clipboard. This covers Chromium's desktop implementations, Firefox's native file paste, and WebKit/Safari on macOS. Current Firefox implementations on macOS and Linux can expose only the first native file, and current non-Cocoa WebKit does not expose native clipboard files. Towk never reads a local path itself: if the clipboard contains a local file reference but the browser withholds the file, the composer consumes the reference, displays an error, and leaves the path out of the message. Drag-and-drop and the picker remain available in that case.

| Asset type | Behavior |
| ---------- | -------- |
| Images | Dimensions are inspected, image URLs can request resized WebP variants, and thumbnails are cached when enabled. |
| Documents and ordinary files | Stored as persisted assets and exposed through signed URLs. |
| Archives | ZIP, 7z, RAR, tar, and compressed archives are accepted as opaque files and are not extracted by Towk. |
| Executables and installers | Direct executable, installer, command-script, WebAssembly, and macro-enabled Office payloads are rejected by browser metadata/content checks and authoritative server checks. |
| Videos | Accepted only when video processing is enabled. |
| Animated GIFs | Uploaded as images when video processing is disabled; routed through video processing when enabled. |
| Avatars, logos, banners | Stored in the same asset system but gated by profile or admin permissions. |
| Link-preview images | Downloaded, converted to a size-limited WebP, and stored in a dedicated quota-bounded NATS object store. |

Default upload limits are 25 MB for non-video files and 100 MB for videos when video processing is enabled.

Each message is limited to 10 attachments. Upload sessions use 512 KiB chunks, expire after 15 minutes while open, and unclaimed completed assets expire after 24 hours. These application bounds complement, but do not replace, the storage quota and request-rate controls of the NATS, S3, reverse-proxy, or WAF layer.

`SERVER_ASSETS` also has a hard 10 GB JetStream quota by default. Configure it with `core.assets.max_store_bytes` or `CHATTO_CORE_ASSETS_MAX_STORE_BYTES`. The limit covers NATS-backed assets and temporary upload chunks; when S3 is enabled, the final bucket still needs its own provider-side quota and monitoring. Lowering the NATS limit preserves existing objects and blocks new writes until usage is back under the limit.

The executable policy is defense in depth, not a malware scanner. Filename extensions and browser MIME types are untrusted hints; the server normalizes bounded metadata and inspects the checksum-verified payload before it creates an asset. Towk deliberately does not unpack archives, so archive-bomb expansion is outside the upload path.

## Permission And Access Model

File attachments require:

- room membership
- `message.post` or `message.post-in-thread`
- `message.attach`

Attachment URLs are signed per user and still check that the signed user can access the room. If a user leaves, is removed, or loses room access, new asset fetches fail even if an old message still contains the URL.

<Aside type="note">
  Signed attachment URLs are bearer capabilities until they expire. Towk keeps their TTL bounded and checks the signed user's current room access on every protected asset fetch.
</Aside>

## Room Files

The room sidebar Files panel lists current accessible attachments from root messages and thread replies. It derives from visible message state, so files disappear when their message is deleted or the attachment is removed.

Use this as a member convenience surface, not as a permanent document-management system. There is no attachment search or media filtering yet.

## Link Previews

Towk fetches link-preview metadata when a user composes a message with a URL. New preview images use the dedicated `LINK_PREVIEW_ASSETS` NATS object store even when ordinary uploads use S3. This isolates remotely fetched content behind a hard quota. A same-ID `SERVER_ASSETS` object link lets older replicas serve new preview images during rolling upgrades and rollback without duplicating bytes; older S3 and legacy `SERVER_ASSETS` preview objects also remain readable after upgrade.

The fetcher is designed to be SSRF-safe, rejects images above 10 million decoded pixels, limits the encoded WebP to 1 MiB, coalesces concurrent requests for the same URL, and runs at most two remote fetches per process. It caches successful and failed lookups for bounded periods and applies shared source-IP and user request budgets. An elected cleanup pass waits for projection replay, preserves images referenced by current messages even when lifecycle metadata is stale or missing after restore, repairs interrupted claims, and removes expired unclaimed or no-longer-referenced objects. Operators do not need a separate preview worker.

## Storage Backend Choice

| Deployment | Recommended backend |
| ---------- | ------------------- |
| Small server with light uploads | NATS ObjectStore, the default. |
| Active community with many files | S3-compatible storage. |
| Video-heavy server | S3-compatible storage plus video processing. |
| Replicated NATS cluster | S3-compatible storage to avoid multiplying large binary storage. |

Switching to S3 affects new assets. Existing NATS-stored assets continue to work because each asset records its own backend.

## Video And Animated GIF Processing

Video processing is optional and disabled by default. When enabled, Towk requires `ffmpeg` and `ffprobe`, stores the original upload, generates a thumbnail, and creates web-friendly MP4 variants based on the source resolution.

Enable it only when the server has CPU and temporary disk space for transcoding:

```toml
[video]
enabled = true
max_concurrent = 2
```

The official Docker image includes the required ffmpeg tools. Host installs need those binaries installed separately.

<Aside type="caution">
  Video processing is currently in-process and best-effort. Towk records durable asset processing state and retries missed unmanifested video assets on boot, but operators should still treat transcoding as CPU-heavy background work.
</Aside>

## Active Document Safety

Uploaded HTML, XHTML, SVG, XML, and XML-derived active document formats are served with a sandbox policy and `nosniff`. S3-backed active documents stream through Towk instead of redirecting directly to object storage so the sandbox headers stay in place.
