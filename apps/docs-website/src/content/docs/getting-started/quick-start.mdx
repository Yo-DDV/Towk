Towk can run as one binary with an embedded NATS data store. The following path
is intended for local evaluation; use the deployment guides for an
Internet-facing server.

<Aside type="note">
  Towk's manual release pipeline produces versioned, checksummed binary archives
  and matching immutable container images. Towk `v0.5.0` predates the canonical
  Towk executable in release archives, so build the reviewed source until a
  later Towk-branded release is available. Pin an exact release or image digest
  for durable deployments. A Homebrew formula is not currently published.
</Aside>

<Steps>

1. **Clone the Towk repository**

   ```bash
   git clone https://github.com/Yo-DDV/Towk.git
   cd Towk
   ```

2. **Install the pinned development toolchain**

   Install [mise](https://mise.jdx.dev/), review `mise.toml`, then run:

   ```bash
   mise trust
   mise install
   mise run build-dev-cli
   ```

3. **Generate a local configuration**

   ```bash
   mkdir -p .local/quick-start
   cd .local/quick-start
   ../../cli/bin/towk init
   ```

   The command creates `towk.toml` with random
   signing, encryption, runtime-token, asset, and NATS secrets. Keep that file
   private and review it before exposing the server.

4. **Start Towk**

   ```bash
   ../../cli/bin/towk run -c towk.toml
   ```

5. **Open the local server**

   Visit [http://localhost:4000](http://localhost:4000). Stop the process with
   `Ctrl-C` when finished.

</Steps>

The local configuration uses embedded NATS. Uploaded files also use NATS unless
you configure S3-compatible storage. Voice and video calls require a separate
[LiveKit](https://github.com/livekit/livekit) server; they can remain disabled
for this evaluation.

## Next steps

Before a durable deployment, decide how you will handle TLS, SMTP, backups,
owner recovery, external NATS, object storage, calls, push notifications, and
monitoring.
